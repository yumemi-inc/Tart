# 内部メモ

コードの背景にある考えを残したいときは、ここに設計メモや仕様メモを置きます。
これは利用者向けのドキュメントではありません。

## 方針

- コードだけでは追いにくい判断理由を残すために使います。
- 形式は軽く保ちます。続けられることを優先します。
- 古くなったメモはすぐ消さず、冒頭の `状態` を `廃止` にして明示します。

## 最低限書くこと

- 何についてのメモか
- 今の結論、または考えている方向
- 反映状況
- 関連する Issue、Pull Request、実装ファイル、テスト

## 使い方

- 1 トピックごとに 1 ファイル追加します。
- 必要なら [`template.md`](./template.md) をコピーして使います。
- ファイル名は自由ですが、迷ったら `YYYY-MM-DD-short-title.md` にします。

## テンプレートの項目

- `タイトル`: 後で一覧したときに見分けやすい短い題名です。
- `状態`: そのメモが今どういう状態かを示します。検討中なら `メモ`、結論が固まっていれば `決定済み`、古くなっていれば `廃止` を使います。
- `更新日`: 最終更新日です。情報の鮮度を判断しやすくします。
- `反映状況`: その結論が実装、テスト、README などにどこまで反映されているかを示します。迷ったら `未反映`、`一部反映`、`反映済み` を使います。
- `関連`: 関連する Issue、Pull Request、実装ファイル、テストへの導線です。
- `背景`: なぜこのメモが必要なのか、どんな背景や問題があるのかを書きます。
- `結論`: 今の結論、または最も有力な方向を書きます。いちばん重要な項目です。
- `補足`: 制約、代替案、実装上の注意など、補足情報を書きます。
- `未解決事項`: まだ決まっていないことや、後で確認が必要なことを書きます。

実運用では `背景` と `結論` が埋まっていれば十分です。ほかの項目は必要なときだけ使えば足ります。
